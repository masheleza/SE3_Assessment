# FinancialStatements.DocumentApi

Document generation and retrieval API (.NET 8). Consumes statement requests from SQS, generates PDFs, persists metadata to SQL Server, caches content in Redis, stores files in S3, and signals completion back via a response queue.

## Responsibilities

- **Event-driven processing** — `SqsDocumentConsumer` (BackgroundService) polls the SQS request queue using long-polling and processes up to 5 messages concurrently.
- **Document generation** — `DocumentStorageService.GenerateStatementAsync` produces the PDF. The scaffold contains a UTF-8 text stub; replace with QuestPDF, iTextSharp, or SSRS for production.
- **Persistent layer** — `DocumentRecord` entities are stored in SQL Server via EF Core. Migrations run automatically on startup.
- **Caching** — Redis cache-aside: generated bytes are stored immediately after creation and served on subsequent reads without hitting S3.
- **Storage** — PDFs are uploaded to S3 with server-side AES-256 encryption under the key pattern `statements/{yyyy}/{MM}/{documentId}.pdf`.
- **Response events** — publishes `DocumentReadyEvent` to the BFF's response queue after success or failure.

Consumers/
  SqsDocumentConsumer.cs        BackgroundService — polls SQS request queue

Infrastructure/
  DbContext/
    DocumentDbContext.cs        EF Core context with index configuration
  Repositories/
    DocumentRepository.cs       IDocumentRepository — upsert and bulk-status update
Migrations/
  *_InitialCreate.cs            EF Core migration — creates the Documents table
  DocumentDbContextModelSnapshot.cs   Current model snapshot
Models/
  DocumentModels.cs             DocumentRecord EF entity
Services/
  DocumentService.cs            Orchestrates cache → storage → DB → SQS response
  DocumentStorageService.cs     S3 upload/download + PDF generation stub

Shared enums (`StatementType`, `DocumentStatus`), domain events (`StatementRequestedEvent`, `DocumentReadyEvent`), and DTOs live in the `FinancialStatements.Models` library. `Models/DocumentModels.cs` contains only the `DocumentRecord` EF entity.

## Processing flow

```
SqsDocumentConsumer receives StatementRequestedEvent
  └─► DocumentService.ProcessRequestAsync
        ├─ Upsert DocumentRecord (Status = Processing)
        ├─ DocumentStorageService.GenerateStatementAsync  →  byte[]
        ├─ DocumentStorageService.UploadAsync             →  S3 key
        ├─ DocumentRepository.UpdateStatusAsync           →  Status = Ready
        ├─ Redis.StringSetAsync                           →  cache (2h TTL)
        └─ SQS.SendMessageAsync                          →  DocumentReadyEvent
```

On failure the record is set to `Status = Failed` and a `DocumentReadyEvent` with `Success = false` is published so the BFF can notify the user.

## Cache-aside read path

```
DocumentService.GetDocumentAsync(documentId)
  1. Redis GET  →  hit: return bytes
  2. DB lookup  →  Status != Ready or no StoragePath: return null
  3. S3 GET     →  return bytes, populate Redis (2h TTL)
```

## Exception handling

`GlobalExceptionHandler` implements `IExceptionHandler` and is registered in `Program.cs` via `AddExceptionHandler<GlobalExceptionHandler>()`. All unhandled exceptions are:

1. Logged at `Error` level with the full exception and request context.
2. Mapped to an HTTP status code.
3. Returned as `ProblemDetails` JSON with a `traceId` field for correlation.

| Exception | Status |
|---|---|
| `InvalidOperationException` | 400 |
| `UnauthorizedAccessException` | 401 |
| `KeyNotFoundException` | 404 |
| All others | 500 |

## Persistent layer

Entity: `DocumentRecord`

| Column | Type | Notes |
|---|---|---|
| `Id` | `uniqueidentifier` | Set by BFF, never DB-generated |
| `UserId` | `nvarchar(256)` | Indexed |
| `AccountId` | `nvarchar(100)` | Composite index with UserId |
| `Type` | `int` | `StatementType` enum |
| `PeriodFrom / To` | `datetimeoffset` | |
| `Status` | `int` | `Pending → Processing → Ready / Failed` |
| `StoragePath` | `nvarchar(1024)` | S3 object key |
| `ErrorMessage` | `nvarchar(2048)` | Populated on failure |
| `CreatedAt / CompletedAt` | `datetimeoffset` | |

EF Core migrations are applied automatically via `db.Database.MigrateAsync()` in `Program.cs`. The initial schema is defined by the `InitialCreate` migration under `Migrations/`, which creates the `Documents` table along with the `IX_Documents_UserId` and `IX_Documents_UserId_AccountId` indexes.

## Replacing the PDF stub

Open `Services/DocumentStorageService.cs` and replace the body of `GenerateStatementAsync`. Example with QuestPDF:

```csharp
public Task<byte[]> GenerateStatementAsync(StatementRequestedEvent request, CancellationToken ct)
{
    var document = Document.Create(container =>
    {
        container.Page(page =>
        {
            page.Content().Text($"Statement for {request.AccountId}");
        });
    });

    using var ms = new MemoryStream();
    document.GeneratePdf(ms);
    return Task.FromResult(ms.ToArray());
}
```

## Configuration (`appsettings.json`)

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Server=localhost,1433;Database=FinancialStatements;User Id=statement;TrustServerCertificate=True",
    "Redis": "localhost:6379"
  },
  "Sqs": {
    "RequestQueueUrl": "..."
  },
  "Storage": {
    "BucketName": "financial-statements-documents"
  }
}
```

The `DefaultConnection` string in `appsettings.json` deliberately omits the SQL password — it is never committed. Supply the password (or the full connection string) out-of-band:

```bash
# Local development — .NET user secrets (stored outside the repo)
dotnet user-secrets set "ConnectionStrings:DefaultConnection" \
  "Server=localhost,1433;Database=FinancialStatements;User Id=statement;Password=<password>;TrustServerCertificate=True"

# Other environments — environment variable (double underscore = config nesting)
export ConnectionStrings__DefaultConnection="...;Password=<password>;..."
```

Configuration layering means the secret/env value overrides the password-less entry in `appsettings.json` at startup.

The response queue URL is not configured here — it arrives per-message inside `StatementRequestedEvent.ResponseQueueUrl`, allowing the BFF to route responses to its own queue without the Document API needing to know about it.

## Running locally

```bash
dotnet run
# Swagger: https://localhost:7002/swagger
```

SQL Server and Redis must be running. On first start, EF Core creates and migrates the database automatically.

## Running EF Core migrations manually

```bash
# From this project directory
dotnet ef migrations add <MigrationName>
dotnet ef database update
```

## Running tests

```bash
dotnet test ../FinancialStatements.DocumentApi.Tests
```

Test coverage includes: `DocumentService` cache-hit/miss paths and processing success/failure flows, `DocumentRepository` upsert and status-update operations (EF Core InMemory), `SqsDocumentConsumer` message processing and delete-on-success behaviour, `DocumentController` action results, and the `GlobalExceptionHandler` status-code mapping and ProblemDetails body.
